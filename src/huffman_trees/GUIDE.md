# Huffman Tree — 手写指南

> 语言：MoonBit  
> 目标：从零手写 Huffman Tree（哈夫曼树）构建与编码流程，理解“带权路径长度最小”的本质，并能实现编码与解码。

---

## 一、核心概念

### 什么是 Huffman Tree

- Huffman Tree（哈夫曼树）是一棵**带权路径长度最小**的二叉树，常用于无损压缩。
- 每个叶子节点代表一个字符（或符号），其权重是该字符出现频率。
- 编码规则通常约定：
	- 向左走记为 `0`
	- 向右走记为 `1`
- 某字符的编码就是从根到该叶子节点的路径比特串。

### 为什么它能压缩

- 高频字符会被放在更靠近根的位置，编码更短。
- 低频字符编码更长。
- 这样整体平均码长下降，压缩效果更好。

### 前缀码（Prefix Code）

- Huffman 编码是前缀码：任意一个字符编码都不是另一个字符编码的前缀。
- 前缀码保证了解码时不需要分隔符，可以从左到右唯一解码。

---

## 二、问题建模

### 输入和输出

- 输入：一段文本，或一个“字符 -> 频率”映射。
- 输出：
	- Huffman 树根节点
	- 每个字符对应的编码表（`Map[Char, String]`）
	- （可选）编码后的 bit 串，和解码函数

### 节点结构设计

建议定义一个树节点结构，至少包含：

- `freq`：该节点权重（频率）
- `ch`：字符（仅叶子节点需要；内部节点可为空）
- `left` / `right`：左右子树

内部节点的 `freq = left.freq + right.freq`。

---

## 三、核心算法：如何构造 Huffman Tree

### 贪心策略

每一步都取出当前频率最小的两个节点合并，直到只剩一个根节点。

这就是 Huffman 的关键贪心思想。

### 为什么要用最小堆

- 你需要反复做“取最小两个节点”的操作。
- 用最小堆可让每次取最小代价为 O(log n)。

如果有 `k` 个不同字符：

- 总体时间复杂度约为 O(k log k)

### 构造流程

```
1. 统计每个字符频率
2. 为每个字符创建一个叶子节点，放入最小堆
3. while 堆中节点数 > 1:
		 a) x = pop_min()
		 b) y = pop_min()
		 c) parent = Node(freq = x.freq + y.freq, left = x, right = y)
		 d) push(parent)
4. 堆中最后剩下的节点就是根节点
```

## 四、手动推演示例

字符频率如下：

```
A: 5, B: 9, C: 12, D: 13, E: 16, F: 45
```

每次取最小两个合并：

```
1) 5 + 9 = 14
2) 12 + 13 = 25
3) 14 + 16 = 30
4) 25 + 30 = 55
5) 45 + 55 = 100  (root)
```

然后从根 DFS：左记 `0`，右记 `1`，即可得到每个字符编码。

注意：当某些频率相同，左右子树交换会得到不同编码，但总压缩率等价。

## 五、生成编码表（Code Table）

树构建完成后，做一次 DFS / 递归遍历：

- 到叶子节点：记录当前路径字符串为该字符编码
- 递归到左子树：路径加 `0`
- 递归到右子树：路径加 `1`

伪代码：

```
build_code(node, path):
	if node is leaf:
		table[node.ch] = path
		return

	if node.left != None:
		build_code(node.left, path + "0")

	if node.right != None:
		build_code(node.right, path + "1")
```

## 六、编码与解码

### 编码（Encode）

- 遍历原始字符串每个字符 `c`
- 查表 `table[c]`
- 拼接成 bit 串

### 解码（Decode）

- 从根出发，读 bit：
	- `0` 走左
	- `1` 走右
- 到叶子时输出该字符，并回到根继续处理下一个 bit

伪代码：

```
decode(bits):
	cur = root
	for b in bits:
		cur = (b == '0') ? cur.left : cur.right
		if cur is leaf:
			output cur.ch
			cur = root
```

## 七、边界情况与易错点

### 1. 只有一个字符

- 例如输入全是 `a`。
- 这时树只有一个节点，常见做法是给它编码 `0`（或 `1`），避免空编码。

### 2. 频率相同

- 多个字符频率一样时，编码结果可能不唯一。
- 为了测试稳定性，可在最小堆比较中加入二级键（例如字符字典序或插入序）。

### 3. 直接拼字符串效率问题

- 编码阶段如果大量 `String` 直接拼接，可能产生性能损耗。
- 可用可变缓冲（如字符数组/byte buffer）后再一次性转字符串。

### 4. 存储开销

- 真正压缩文件时，不能只存 bit 串，还要存“如何重建树”的信息（频率表或树结构）。
- 练习题中通常先关注算法正确性。

---

## 八、复杂度分析

设不同字符个数为 `k`，文本长度为 `n`：

- 统计频率：O(n)
- 构建最小堆：O(k)
- 合并构树：O(k log k)
- 生成编码表（遍历树）：O(k)
- 编码文本：O(n)
- 解码文本：O(编码后 bit 数)

整体瓶颈通常在 O(n + k log k)。

## 九、实现顺序建议（MoonBit）

```
1. 定义 Node 结构（freq/ch/left/right）
2. 写 frequency_count：文本 -> 频率表
3. 准备最小堆（可复用已有 heap 包，或手写一个最小堆）
4. 写 build_huffman_tree
5. 写 build_code_table（DFS）
6. 写 encode(text, table)
7. 写 decode(bits, root)
8. 补齐边界情况：空串、单字符、重复频率
9. 写测试：可逆性（decode(encode(x)) == x）与前缀码性质
```

## 十、测试建议

至少覆盖以下测试：

- 基本功能：普通字符串可以正确编码再解码。
- 空输入：空字符串返回空编码，解码也为空。
- 单字符输入：如 `"aaaaaa"`。
- 混合频率：高频字符编码应不长于低频字符（统计意义下）。
- 前缀码验证：任意两个字符编码互不为前缀。

一个很实用的核心断言：

```
decode(encode(text, table), root) == text
```

只要这一条在多组样例下稳定成立，说明你的主流程基本正确。
